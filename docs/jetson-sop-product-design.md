# AI 视觉 SOP 合规检测系统产品设计文档

## 1. 产品概述

### 1.1 产品名称

AI 视觉 SOP 合规检测系统

### 1.2 产品定位

本产品是一套基于 AI 视觉识别的 SOP 合规检测系统，用于在检测工位实时识别操作员是否按照标准流程完成产品检测动作，并将每一步检测状态、检测结果和异常原因可视化展示。

当前需求版本的 SOP 流程仅包含三个步骤：

1. 正面
2. 反面
3. 整体

系统通过摄像头采集工位画面，结合目标检测、手部关键点识别、检测区域 ROI 和 SOP 状态机，判断正面、反面、整体三个检测动作是否符合规范。

### 1.3 目标用户

- 工厂产线管理人员
- 质检人员
- 检测工位操作员
- 工业视觉系统集成商
- 边缘 AI 设备方案商

### 1.4 核心价值

- 将人工 SOP 巡检转为实时视觉合规检测
- 降低漏检、错检、跳步骤和不规范操作风险
- 形成检测过程留痕，支持质量追溯
- 通过统一状态机展示每一步检测结果
- 为后续对接 MES、ERP、质检报表提供基础数据

## 2. SOP 步骤设计

### 2.1 SOP 步骤范围

当前版本仅包含以下三个 SOP 步骤：

| No | 步骤 | 说明 |
| --- | --- | --- |
| 1 | 正面 | 检测产品正面是否按 SOP 要求完成展示、停留、拍摄或识别 |
| 2 | 反面 | 检测产品反面是否按 SOP 要求完成展示、停留、拍摄或识别 |
| 3 | 整体 | 在正面和反面均符合 SOP 规范后，检测产品整体状态是否符合规范 |

> 注：用户原需求中“方面”按上下文理解为“反面”，文档统一使用“反面”。

### 2.2 步骤关系

三个步骤必须按照固定顺序执行：

```text
正面
  ↓
反面
  ↓
整体
  ↓
输出最终结果
```

### 2.3 步骤推进规则

- 任务创建后，所有步骤默认状态为 `Idle`。
- 点击或触发开始检测后，当前步骤进入 `Running`。
- 当前步骤检测符合 SOP 规范时，状态变为 `Pass`。
- 当前步骤检测不符合 SOP 规范时，状态变为 `Failed`。
- 正面步骤 `Pass` 后，反面步骤才允许进入 `Running`。
- 反面步骤 `Pass` 后，整体步骤才允许进入 `Running`。
- 若正面或反面任一步骤为 `Failed`，整体步骤不应直接判定通过，需提示重新检测。

## 3. 检测结果状态逻辑

### 3.1 状态定义

| 状态 | 触发条件 | 含义 | UI 表现 |
| --- | --- | --- | --- |
| Idle | 尚未开始检测 | 未开始检测 | 深蓝 / 灰色 |
| Running | 用户开始当前步骤检测 | 开始检测 / 正在检测 | 蓝色 |
| Pass | 当前步骤检测符合 SOP 规范 | 检测成功 | 绿色 |
| Failed | 当前步骤检测不符合 SOP 规范 | 检测失败 | 红色 |

### 3.2 状态机规则

```text
Idle
  ↓ 开始检测
Running
  ├─ 检测符合 SOP → Pass
  └─ 检测不符合 SOP → Failed
```

### 3.3 最终结果规则

| 正面 | 反面 | 整体 | 最终结果 |
| --- | --- | --- | --- |
| Pass | Pass | Pass | Pass |
| Failed | 任意 | 任意 | Failed |
| Pass | Failed | 任意 | Failed |
| Pass | Pass | Failed | Failed |
| Idle / Running | 任意 | 任意 | 检测中 |

### 3.4 状态显示文案

| 状态 | 展示文案 |
| --- | --- |
| Idle | 未开始检测 |
| Running | 正在检测 |
| Pass | 检测成功 |
| Failed | 检测失败 |

## 4. 失败提示规则

### 4.1 正面步骤失败提示

当“正面”步骤检测失败时，说明提示固定为：

```text
本次操作未按照SOP规范检测
```

### 4.2 反面步骤失败提示

当“反面”步骤检测失败时，说明提示固定为：

```text
本次操作未按照SOP规范检测
```

### 4.3 整体步骤失败提示

当进入“整体”步骤时，如果正面或反面检测操作不符合 SOP 规范，则整体步骤说明提示为：

```text
由于正面（反面）检测操作不符合SOP规范，请重新检测。
```

推荐按具体失败来源生成更明确的提示：

| 前置失败步骤 | 整体步骤提示 |
| --- | --- |
| 正面 Failed | 由于正面检测操作不符合SOP规范，请重新检测。 |
| 反面 Failed | 由于反面检测操作不符合SOP规范，请重新检测。 |
| 正面、反面均 Failed | 由于正面（反面）检测操作不符合SOP规范，请重新检测。 |

## 6. 功能需求

### 6.1 摄像头接入

#### 功能说明

接入工位摄像头，实时采集检测操作画面。

#### 需求

- 支持 USB 摄像头
- 支持 RTSP / ONVIF 网络摄像头
- 支持 720p / 1080p 分辨率
- 支持帧率配置
- 支持实时预览
- 支持摄像头断线重连
- 支持画面镜像、旋转和裁剪

### 6.2 ROI 检测区域配置

#### 功能说明

设置工位有效检测区域，减少背景干扰。

#### 需求

- 支持在视频画面上绘制矩形或多边形 ROI
- 支持保存 ROI 配置
- 支持每个工位独立配置 ROI
- 仅当目标进入 ROI 时参与 SOP 判定

### 6.3 AI 视觉检测

#### 功能说明

识别产品、手部、检测姿态和当前展示面，判断当前操作是否符合正面、反面、整体三个 SOP 步骤要求。

#### 建议识别类别

| 类别 | 说明 |
| --- | --- |
| product_front | 产品正面 |
| product_back | 产品反面 |
| product_whole | 产品整体 |
| left_hand | 左手 |
| right_hand | 右手 |
| roi | 检测区域 |

#### 检测输出

```json
{
  "class_name": "product_front",
  "confidence": 0.93,
  "bbox": [120, 80, 360, 280],
  "timestamp": 1778344800000
}
```

### 6.4 手部关键点识别

#### 功能说明

识别操作员左右手和手部关键点，用于判断检测动作是否真实发生，而不是仅凭产品静态出现在画面中判定通过。

#### 需求

- 识别左右手
- 输出手部检测框
- 输出关键点骨架
- 判断手部是否进入 ROI
- 判断手部与产品是否存在有效交互
- 判断产品正面 / 反面 / 整体展示动作是否满足停留时间要求

### 6.5 SOP 状态机

#### 功能说明

状态机负责把 AI 识别结果转化为三个业务步骤状态。

#### 状态定义

```text
Idle     未开始检测
Running  正在检测
Pass     检测成功
Failed   检测失败
```

#### 推进规则

- 任务开始前，正面、反面、整体均为 `Idle`。
- 开始正面检测后，正面进入 `Running`。
- 正面检测成功后，正面变为 `Pass`，反面允许进入 `Running`。
- 正面检测失败后，正面变为 `Failed`，说明提示“本次操作未按照SOP规范检测”。
- 反面检测成功后，反面变为 `Pass`，整体允许进入 `Running`。
- 反面检测失败后，反面变为 `Failed`，说明提示“本次操作未按照SOP规范检测”。
- 整体步骤需要检查正面和反面是否均为 `Pass`。
- 如果正面或反面存在 `Failed`，整体步骤提示“由于正面（反面）检测操作不符合SOP规范，请重新检测。”

#### 判定示例

```json
{
  "step_id": 1,
  "name": "正面",
  "required_objects": ["product_front"],
  "required_hand_interaction": true,
  "stable_frame_count": 5,
  "min_confidence": 0.8,
  "timeout_seconds": 30,
  "failed_message": "本次操作未按照SOP规范检测"
}
```

### 6.6 实时 UI 展示

#### 功能说明

前端实时展示摄像头画面、检测框、SOP 状态、当前日志、失败提示和最终结果。

#### 需求

- Canvas 绘制 ROI、检测框和手部关键点
- 右侧展示 SOP 状态表
- 底部展示三个步骤状态卡片
- 顶部展示项目、作业员、耗时、最终结果
- 支持 Pass / Failed 高亮显示
- 支持失败提示文案展示
- 支持检测中、检测成功、检测失败等状态切换

### 6.7 检测记录保存

#### 功能说明

每次检测流程结束后生成检测记录。

#### 数据字段

| 字段 | 说明 |
| --- | --- |
| task_id | 检测任务 ID |
| sop_id | SOP 模板 ID |
| workstation_id | 工位 ID |
| operator_id | 作业员 ID |
| start_time | 开始时间 |
| end_time | 结束时间 |
| duration | 总耗时 |
| result | Pass / Failed |
| failed_step | 失败步骤 |
| failed_message | 失败提示 |
| snapshots | 关键帧截图 |
| logs | 检测日志 |

### 8.4 关键交互

- 点击“开始检测”后，正面步骤进入 `Running`。
- 当前步骤完成后，系统自动或手动进入下一步骤。
- 点击 SOP 步骤可查看该步骤判定规则。
- 点击失败日志可查看对应关键帧。
- 点击重新检测可重置当前任务，三步状态恢复为 `Idle`。
- 管理员可进入配置页调整 ROI、阈值和失败提示。

## 10. 技术方案

### 10.1 系统架构

```text
摄像头
  ↓
视频采集服务
  ↓
AI 推理服务
  ↓
目标检测 / 手部关键点识别
  ↓
SOP 状态机
  ↓
实时消息推送
  ↓
Web 监控界面
  ↓
日志与结果存储
```

### 10.2 推荐技术栈

#### 前端

- React / Vue
- Canvas / WebGL 绘制检测框
- WebSocket 接收实时检测结果
- 表格组件展示 SOP 状态
- ECharts 展示统计报表

#### 后端

- Python FastAPI
- WebSocket
- REST API
- PostgreSQL / MySQL
- Redis 缓存实时状态

#### AI 推理

- YOLOv8 / YOLOv10 / RT-DETR：目标检测
- MediaPipe Hands / 自训练手部关键点模型：手部识别
- ONNX Runtime / TensorRT：推理加速
- OpenCV：视频采集与图像处理

#### 边缘部署

- NVIDIA Jetson Nano / Xavier / Orin
- 本地摄像头接入
- 本地模型推理
- 局域网 Web 管理界面
- 离线缓存检测记录

## 12. 验收标准

### 12.1 基础功能验收

- 系统可稳定接入摄像头并显示实时画面。
- 系统仅展示三个 SOP 步骤：正面、反面、整体。
- SOP 表格可正确显示 `Idle`、`Running`、`Pass`、`Failed` 状态。
- 底部步骤状态条与右侧 SOP 表格保持一致。
- 正面检测成功后状态显示 `Pass`。
- 反面检测成功后状态显示 `Pass`。
- 整体检测成功后状态显示 `Pass`。
- 正面或反面检测失败时，对应步骤状态显示 `Failed`，说明为“本次操作未按照SOP规范检测”。
- 正面或反面检测失败后，整体步骤提示“由于正面（反面）检测操作不符合SOP规范，请重新检测。”

### 12.2 性能验收

- 实时画面端到端延迟小于 500ms。
- 单帧推理耗时小于 100ms。
- 正常光照下目标检测准确率大于 95%。
- 连续运行 8 小时无崩溃。
- 摄像头断线后可自动重连。

### 12.3 业务验收

- 能发现正面检测不规范。
- 能发现反面检测不规范。
- 能在前置检测失败时阻止整体步骤误判通过。
- 能记录每个检测任务的开始、结束和步骤耗时。
- 能查看每个失败任务的失败步骤、失败提示和关键帧。
- 能支持至少一个检测工位稳定运行。

## 13. 复刻开发建议

建议按以下顺序实现：

1. 先完成单工位实时视频预览和检测框绘制。
2. 接入目标检测模型，识别产品正面、反面和整体状态。
3. 接入手部检测，增加操作交互判断。
4. 实现固定三步 SOP 状态机。
5. 完成主监控页面，还原左侧实时画面、右侧状态表、底部状态条布局。
6. 增加失败提示规则和检测记录保存。
7. 再扩展 SOP 可配置、多工位和后台管理能力。

## 14. 总结

该产品的核心不是单纯做物体识别，而是把目标检测、手部行为识别、ROI 约束和 SOP 状态机组合起来，实时判断正面、反面、整体三个检测动作是否按标准流程完成。

复刻开发时应优先保证以下三点：

1. SOP 步骤严格限定为正面、反面、整体。
2. 状态机严格使用 `Idle`、`Running`、`Pass`、`Failed` 四种状态。
3. 失败提示文案必须与业务规则一致，尤其是正面 / 反面失败和整体步骤前置失败提示。
